I'm building a local-only web application for a clinic's patient management system. The application will run on two laptops connected via a local network (LAN) and won't use any paid services or external APIs. All data will be stored in a local database on the hard drive, specifically an SQLite database file.

Here are the key functionalities of the three main pages:

### 1. Reception Page

* **New Patient Registration:** The receptionist enters a new patient's details (Full Name, Date of Birth, Address, Contact Number, Gender, and assigned Consultant). A unique registration number is auto-generated upon registration.
* **Returning Patient:** The receptionist can enter an existing patient's registration number. This automatically populates all fields except the consultant, which can be updated if needed.
* **Patient Search:** A search function allows the receptionist to find patients by name or contact number. The search should return exact or similar matches from the database.
* **Database:** Patient details are stored in a database file on the local hard drive.

### 2. Consultant Page

* **Consultant Selection:** A dropdown menu at the top-right allows the consultant to select their name. This populates the page with patients assigned to them. The list of consultants for this dropdown is populated from a specific table within the local database.
* **Patient Queue:** This section displays a list of patients waiting for consultation, in a chronological order based on their registration time (first patient registered is at the top). Each entry shows the patient's name, registration number, and registration time.
* **Patient Information:** Clicking a patient in the "Patient Queue" displays their full details in this section, including all information entered by the receptionist except for the contact number and address. It should also show the patient's date of birth and their current age. Below this, the last five visit dates for that patient are displayed.
* **Consultation Complete:** After the consultation, the consultant clicks a "Consultation Complete" button. This moves the patient from the "Patient Queue" to the "Completed Queue."
* **Completed Queue:** This section lists patients whose consultations are finished. It has the same format as the "Patient Queue."
* **Reset Button:** The "Reset" button clears the "Patient Information" section and returns to the default state, displaying the information of the patient who was last selected before the reset.
* **Printing:** A "Print" button opens a print window with a summary of the patient's information (registration number, name, age, gender, and visitation date/time) in the top-right corner, followed by five lines for the consultant's notes.
* **Queue Interaction:** The consultant can click on any patient in either the "Patient Queue" or "Completed Queue" to view their details in the "Patient Information" section.

### 3. Report Page

* **Report Generation:** This page allows the user to input a **Start Date** and an **End Date** to generate a summary report.
* **Report Content:** The generated report, displayed on the page or as a downloadable file, will first show the total number of consultations completed within the specified date range. This will be followed by a detailed list of each patient visit, including their registration number, name, and visit date/time, all extracted from the local database.

I need a full-stack web application developed using **Python, Flask, and SQLite** for the database. The front-end should be simple HTML/CSS/JavaScript. Please provide the code for all necessary files (app.py, templates, static files) to create this functional system.
